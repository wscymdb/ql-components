# EditableTreeTable 可编辑树形表格

`EditableTreeTable` 组件是一个标准的 **Editable Tree Grid（可编辑树形网格）**，专为多级树状结构的实时编辑场景设计。

## 依赖与选型说明

本组件底层深度依赖了 `antd`、`@ant-design/pro-components` 以及 `@ant-design/icons`。

- **为什么选择使用 ProComponents？**
  `EditableTreeTable` 内部基于 ProComponents 的 `EditableProTable` 进行封装。ProComponents 已经在底层针对 Ant Design 的 `Form`（表单）与 `Table`（表格）做好了极其成熟的深度二次封装与值映射。如果仅使用原生 `antd` 基础组件去白手起家实现一套“实时编辑同步、嵌套树节点新增与级联删除、统一表单校验”的复杂组件，其开发成本与后续边缘 Case 的维护精力都过高。因此，借力 ProComponents 可以极大提高组件的产出效率与稳定性，避免重复造轮子。

## 典型场景

- **可视化 JSON Schema 编辑器 (Visual JSON Schema Editor)**：
  适用于 API 接口管理平台（如 YApi、Swagger、Apifox），针对 API 响应数据定义（Response Schema）提供多级嵌套配置。
- **递归数据结构定义器 (Recursive Data Structure Definer)**：
  支持对 `Object` , `Array` 等复杂类型进行树状结构的无限递归嵌套与定义。
- **分层数据集批量/就地编辑器 (Hierarchical Dataset Mass/In-place Editor)**：
  支持在表格视图中直接编辑复杂嵌套数据，具备高性能的实时输入体验。

## 受控模式 

**一般业务场景下不需要使用此模式。** 组件库预留了受控属性，以应对特殊的跨组件联动或外部状态干预需求。

例如在以下场景中可能需要受控：
1. **外部控制状态**：在表格外部放置了辅助按钮，需要实现“一键展开所有行”、“一键折叠所有行”、“一键让所有行进入编辑态”或“取消所有编辑”等操作。
2. **初始化条件渲染**：需要根据外部的其他过滤条件或业务状态，在表格初次渲染时强制让某些特定的行默认展开或默认处于编辑激活状态。

## 树数据预处理工具 (Key 自动补全)

在只读模式或常规可编辑模式下，如果后端接口返回的复杂嵌套数据中缺少唯一标识符字段（例如 `id` 或 `key`），由于：
1. **展开符号默认隐藏**：组件默认将 `showExpandIcon` 设为 `false`（即隐藏展开图标）。
2. **唯一标识缺失导致闭合**：底层状态管理器无法解析每行有效的 Key，因此所有嵌套节点都会默认处于折叠状态。

这两种表现结合在一起，在视觉上就会呈现出**子节点完全不可见且无任何展开操作入口**的现象。

为了解决该痛点并恢复正常的树展开交互，组件库直接导出了 `fillTreeKeys` 递归预处理工具函数，可以在不污染原始引用的情况下，自动为节点遍历补齐缺失的唯一键：

## 动态新增与数量限制

在复杂的接口或 Schema 定义业务中，您往往需要对树节点的新增行为做出限制：
1. **全局/根节点限制**：限制最大参数总数。例如当根节点数量达到上限时，禁用底部按钮并展示 Tooltip 提示。
2. **行级联动限制**：当行类型为 `array` 时，仅允许添加 1 个子节点（模拟数组的 item 类型定义）；当行类型为 `object` 时可以添加多个子节点；普通基本类型不允许添加子节点。

## 数据防丢失与清洗校验

在类似 JSON Schema 编辑器的场景中，常会存在关于数据编辑生命周期的冲突问题：
- **误操作保护**：若用户已在 `object` 下配置了较多子字段，却不小心在下拉框中选错类型改为 `string`。如果立刻强行擦除其 `children` 数据，用户的误触恢复代价将极高。
- **数据规范性**：`string` 类型的节点如果最终仍携带着 `children` 数组并发送给后端，将属于格式不规范的脏数据。

本示例演示了如何在不修改组件库源码的前提下，通过巧妙的组合机制完美解决这二者：
1. **隐藏展开按钮（防丢失）**：使用 `showExpandIcon` 属性。若节点的类型变为了基本类型，自动隐藏其展开箭头，使其在视觉上关闭子项并静默保存数据；当再次切回 `object` 时展开箭头重现，原先配置的子参数完好无损。
2. **表单提交前清洗（数据规范）**：在表单最终提交时，在业务层使用递归函数清洗数据结构，彻底剔除所有基本类型下残留的 `children` 脏数据。
